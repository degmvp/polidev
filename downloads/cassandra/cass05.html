Apache Cassandra – LAB 05
Modelagem Correta

Este laboratório consolida o conceito mais importante do Cassandra: modelar os dados a partir da consulta.

Objetivo do LAB
  • Entender por que Cassandra não é relacional
  • Evitar índices e materialized views
  • Criar tabelas por padrão de acesso
  • Aplicar modelagem manual correta

1️⃣ Erro Comum: Pensar como SQL

Tentar adaptar modelos relacionais ao Cassandra gera:

  • Índices ineficientes
  • Views bloqueadas
  • Consultas inválidas

2️⃣ Erro Real: Materialized View Bloqueada

InvalidRequest: Tables cannot have more than 0 materialized views,
failed to create materialized view clientes_por_cidade
Causa:
Materialized Views são limitadas e não recomendadas no Astra DB.

3️⃣ Solução Correta: Tabela Manual

Criar manualmente a tabela para a consulta desejada. 

CREATE TABLE clientes_por_cidade (
    cidade text,
    id UUID,
    nome text,
    email text,
    PRIMARY KEY (cidade, id)
);
✔ Essa é a abordagem correta no Cassandra

4️⃣ Duplicação de Dados

Duplicar dados é normal no Cassandra.

  • Mesmos dados em múltiplas tabelas
  • Cada tabela atende uma consulta
  • Consistência eventual

5️⃣ Pipeline de Escrita

Em aplicações reais:

  • Aplicação grava em várias tabelas
  • Mesmo evento → múltiplas tabelas
  • Leitura sempre simples e rápida

6️⃣ Exemplo Prático

-- tabela principal
INSERT INTO clientes (id, nome, email, cidade)
VALUES (uuid(), 'Ana Silva', 'ana@email.com', 'Recife');

-- tabela por cidade
INSERT INTO clientes_por_cidade (cidade, id, nome, email)
VALUES ('Recife', uuid(), 'Ana Silva', 'ana@email.com');

7️⃣ Regra de Ouro

NUNCA pergunte:

"Como faço JOIN no Cassandra?"

Pergunte:

"Qual tabela eu crio para essa consulta?"

8️⃣ Boas Práticas de Modelagem

  • 1 consulta = 1 tabela
  • Evite índices
  • Evite materialized views
  • Planeje antes de criar

9️⃣ Encerramento do LAB

  • Você modelou corretamente no Cassandra
  • Entendeu limitações reais
  • Adotou o mindset NoSQL
